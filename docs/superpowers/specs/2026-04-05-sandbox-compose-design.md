# Sandbox Compose Refactor Design

## Summary

Refactor the sandbox tooling from an imperative Docker wrapper into a
compose-backed, Effect-native sandbox platform with a strict configuration
contract and a clearer CLI.

The new shape keeps `app`, `api`, and `postgres` as separate services because
that matches the production boundary, but each sandbox is managed as a single
Docker Compose project rather than three unrelated containers. The sandbox CLI
becomes the only supported entry point for sandbox lifecycle operations and
refuses to start when required configuration or host tooling is missing.

The source of truth for shared configuration remains the repository `.env`
files. Sandbox-only runtime values such as sandbox name, service URLs, internal
database URL, host bindings, and Better Auth callback origins are computed by
the sandbox layer and injected as compose overrides.

## Goals

- manage each sandbox as a single Docker Compose project
- make sandbox setup difficult to misuse and quick to diagnose when it fails
- keep the sandbox service boundaries aligned with production deployment
- load shared configuration from the repository `.env` files
- compute sandbox-specific values centrally and override `.env` safely
- support an explicit sandbox name as well as worktree-derived naming
- keep portless working as part of the sandbox flow
- move sandbox domain and configuration logic to Effect-based modules
- provide clear lifecycle, validation, and logging behavior
- fit cleanly into the Turborepo task model

## Non-Goals

- collapsing app, api, and postgres into a single runtime container
- changing production deployment architecture in this pass
- preserving the existing hand-built `docker run` implementation
- making sandbox startup succeed with partial or guessed configuration
- building a generic secrets platform beyond the current `.env` contract
- introducing backward-compatibility shims for the current sandbox internals

## Current Problems

The current implementation spreads sandbox concerns across low-level CLI code:

- `packages/sandbox-cli/src/runtime.ts` manually creates networks, volumes,
  containers, aliases, ports, env entries, and health loops
- env handling is mixed into runtime orchestration and currently relies on a
  combination of inherited `process.env` values and sandbox defaults
- the sandbox record shape is lightweight, but the actual runtime contract is
  implicit and hard to reason about
- the user experience is centered on container plumbing rather than sandbox
  intent
- the architecture is awkward for multiple worktrees because a sandbox is not
  represented as one inspectable compose project
- logging is mostly plain console output instead of Effect-backed structured
  logs and typed failures

## Core Decisions

The following decisions are fixed for this design:

1. each sandbox is one Docker Compose project with `app`, `api`, and
   `postgres` services
2. shared env values continue to come from the repo `.env` files
3. sandbox-only runtime values are always generated by the sandbox layer and
   override `.env`
4. `sandbox-core` becomes an Effect-native package that owns the domain model,
   config contract, tagged errors, URL computation, and compose/runtime spec
5. `sandbox-cli` becomes a thinner Effect CLI around `sandbox-core`
6. the CLI supports explicit naming with `--name` and still supports
   worktree-derived defaults
7. startup fails fast if required env values, host tools, or sandbox setup are
   missing or inconsistent
8. portless remains part of the sandbox flow rather than an external manual
   prerequisite
9. Turborepo remains the execution entry point for app and api dev services

## Architecture

### Package Responsibilities

#### `packages/sandbox-core`

`sandbox-core` becomes the source of truth for sandbox state and behavior at the
domain boundary.

Responsibilities:

- sandbox identity derivation
- sandbox naming rules and explicit-name validation
- compose project naming
- service hostnames and user-facing URL generation
- runtime port planning
- environment loading and validation with `Config`
- runtime override generation for sandbox-specific values
- tagged domain errors and user-facing diagnostics
- structured Effect logging helpers
- serialization boundaries for persisted sandbox metadata

It should expose Effect-based APIs and typed models rather than a handful of
stateless helpers.

#### `packages/sandbox-cli`

`sandbox-cli` becomes the orchestration shell around `sandbox-core`.

Responsibilities:

- parse commands and flags
- resolve current repo and worktree context
- request a runtime spec from `sandbox-core`
- write generated compose env artifacts to sandbox state
- invoke `docker compose` and portless commands
- reconcile persisted sandbox metadata with live runtime state
- render high-signal CLI output

It should stop owning business logic about what a valid sandbox is.

### Compose Model

Keep a single checked-in compose file in the repo, owned by the sandbox package.
The CLI always invokes it with an explicit compose project name:

- `docker compose --project-name <sandbox-name> --env-file <generated-file> ...`

Each sandbox project contains:

- `postgres`
- `api`
- `app`

This gives one inspectable lifecycle unit per sandbox while preserving the
production-like service split.

The compose file should define:

- service images or build targets
- named volumes
- network wiring
- health checks
- mounts needed for worktree development
- command entrypoints for Turborepo-backed dev processes

The compose file should not be rewritten per sandbox. Runtime variance comes
from compose project naming plus generated env overrides.

### Environment Model

Sandbox configuration is split into three layers.

#### Layer 1: Base Shared Environment

Load from the repository `.env` files using the same resolution model the app
already expects. This is the source of truth for shared secrets and integration
settings.

Examples:

- email sender identity
- Resend API key when auth email delivery is enabled
- any shared application secrets not derived by the sandbox itself

#### Layer 2: Validated Sandbox Contract

Define an explicit Effect `Config` contract for the env values the sandbox
requires from layer 1.

Examples:

- auth sender values required by `apps/api`
- any mandatory Better Auth shared secret inputs that are not sandbox-generated
- any integration credentials that the services truly require in sandbox mode

This layer must not silently fill in placeholder values for required config.
If a required value is missing, startup fails before any compose action runs.

Failure messages must include:

- which variable is missing or invalid
- why the sandbox needs it
- where the value should come from
- the next fix the developer should make

#### Layer 3: Computed Sandbox Overrides

Sandbox-only values are generated per sandbox and override any conflicting `.env`
value.

Examples:

- `SANDBOX_ID`
- explicit sandbox name and hostname slug
- `TASK_TRACKER_SANDBOX`
- `HOST`
- `PORT`
- `DATABASE_URL`
- `BETTER_AUTH_BASE_URL`
- `AUTH_ORIGIN`
- `VITE_AUTH_ORIGIN`
- internal service URLs needed between compose services

This resolves the Better Auth URL concern cleanly: the repo `.env` can describe
shared configuration, while sandbox topology is always computed and forced by
the sandbox runtime.

### Sandbox Identity

Sandbox identity should support both inferred and explicit forms.

Default behavior:

- derive from repo root plus worktree path
- preserve a human-friendly slug when possible

Explicit behavior:

- `sandbox up --name <name>`
- `sandbox status --name <name>`
- `sandbox down --name <name>`
- `sandbox logs --name <name>`
- `sandbox url --name <name>`

Rules:

- names must be hostname-safe
- names must map deterministically to compose project names and portless aliases
- explicit names should be persisted so later commands resolve the same sandbox
- explicit names should not depend on current worktree naming conventions

This is especially important for hosted agents and worktrees whose directory
name does not reflect the intended sandbox identity.

### State Model

Persist sandbox metadata in a registry under the user sandbox state directory,
but make the stored shape reflect the new runtime model.

Suggested persisted state:

- sandbox id
- sandbox name
- hostname slug
- compose project name
- repo root
- worktree path when applicable
- generated port bindings
- timestamp metadata
- last known status

The state record should not attempt to duplicate all compose internals. It
should store enough information to reconnect to the runtime, present useful
status output, and clean up deterministically.

### Portless Integration

Portless remains part of sandbox startup.

Rules:

- verify the `portless` binary is available before startup
- ensure the proxy is running on the configured sandbox proxy port
- register aliases derived from the sandbox name
- surface both preferred portless URLs and loopback fallback URLs in output

Preferred behavior:

- if portless setup succeeds, show the friendly URLs as primary
- if alias registration fails but containers are otherwise healthy, keep the
  sandbox running and surface an explicit degraded warning with exact next steps

The CLI should never leave the user guessing whether they should use the
friendly hostname or the loopback URL.

### Logging And Diagnostics

Sandbox logging should use Effect logging instead of ad hoc console strings.

Requirements:

- lifecycle events emit structured logs
- preflight validation failures are tagged and rendered clearly
- compose command failures include the command context and stderr summary
- readiness failures include service-level health state
- log output remains readable for humans in the terminal

The CLI should still print a concise human summary, but it should be backed by
typed failures and structured Effect logs so debugging is consistent.

## CLI UX

The CLI becomes the supported interface for sandbox lifecycle operations.

### Commands

- `sandbox up [--name <name>]`
- `sandbox down [--name <name>]`
- `sandbox status [--name <name>]`
- `sandbox logs [--name <name>] [service]`
- `sandbox url [--name <name>]`
- `sandbox list`

### `up` Behavior

`up` should:

1. resolve worktree and naming context
2. load and validate shared env
3. compute sandbox runtime overrides
4. verify host prerequisites
5. write generated compose env artifacts
6. ensure portless proxy and aliases
7. run compose startup
8. wait for service health
9. persist sandbox state
10. print a concise ready summary

The command must fail before step 6 if the env contract is incomplete.

### `down` Behavior

`down` should:

- resolve the sandbox by explicit name or current worktree
- remove portless aliases
- stop and remove the compose project
- remove persisted registry state for that sandbox
- leave enough context in the output that the user knows what was cleaned up

### `status` Behavior

`status` should:

- inspect the compose project state
- reconcile persisted metadata with live runtime state
- report `ready`, `degraded`, or `stopped`
- show URLs, project name, and any missing pieces

### `logs` Behavior

`logs` should:

- default to aggregated project logs
- support an optional service selector such as `app`, `api`, or `postgres`
- keep output direct and readable without burying the user in orchestration
  detail

### `list` Behavior

`list` should show all known sandboxes with:

- sandbox name
- compose project name
- status
- worktree path when present
- primary app and api URLs

## Turborepo Integration

Turborepo remains the package task runner and sandbox services should continue
to use package-owned scripts rather than root-owned task logic.

Requirements:

- app and api keep package-local `sandbox:dev` scripts
- root scripts remain simple `turbo run ... --filter=...` delegates
- compose service commands should execute those package scripts, not ad hoc
  root-level shell pipelines
- any sandbox-specific Turborepo env declarations should stay explicit in
  `turbo.json`

This keeps the sandbox flow aligned with the existing monorepo model instead of
creating a separate orchestration path that bypasses Turborepo.

## Error Handling

Model failures as tagged errors with focused user guidance.

Important categories:

- missing or invalid env config
- missing host tools such as Docker or portless
- compose startup failure
- port binding conflicts
- sandbox name conflict
- health check timeout
- sandbox not found

Error output should prefer direct diagnosis over stack traces.

Examples of expected guidance:

- missing env var and which `.env` file should define it
- invalid sandbox name and allowed format
- compose project failed and the next `sandbox logs` command to run
- health timeout and which service is still unhealthy

## Testing Strategy

Follow TDD for the refactor.

### `sandbox-core`

Add focused unit tests for:

- env contract decoding and validation failures
- sandbox naming and explicit-name validation
- compose project name derivation
- runtime URL computation
- sandbox override generation
- persisted state encoding and decoding boundaries

### `sandbox-cli`

Add orchestration tests for:

- `up` refusing to start on missing env
- `up --name` overriding inferred sandbox naming
- compose command argument generation
- portless alias command generation from explicit names
- `status`, `down`, and `logs` lookup behavior for named sandboxes
- degraded mode output when alias registration fails

### Integration Confidence

At minimum, verification for implementation should include:

- package tests for `sandbox-core`
- package tests for `sandbox-cli`
- type-checking for both packages
- one end-to-end local sandbox boot against compose in a real worktree

## Migration Notes

This is a greenfield-friendly refactor, so simplicity should win over
compatibility.

Migration posture:

- remove the hand-built `docker run` orchestration instead of preserving it in
  parallel
- reshape sandbox records as needed for the compose model
- prefer renaming or reorganizing modules when that clarifies the design
- avoid carrying forward placeholder defaults that weaken configuration safety

## Open Questions Resolved

- shared env source: use the repository `.env` files
- Better Auth URLs: compute and inject sandbox-specific overrides at runtime
- sandbox shape: one compose project per sandbox with app/api/postgres services
- sandbox naming: support both inferred names and explicit `--name`
- package direction: make `sandbox-core` Effect-native

## Implementation Outline

The implementation plan should break the work into at least these areas:

1. move sandbox domain/config/runtime spec logic into Effect-based
   `sandbox-core`
2. add the checked-in compose file and generated env artifact flow
3. refactor the CLI to use compose orchestration plus Effect logging
4. update status, list, logs, and teardown around compose project state
5. tighten Turborepo task wiring and verification coverage
